# Shell Voyager Plan

## Context
The goal is to create a system that learns from shell commands, acting as an automatic cheat sheet of frequently used and successful commands per working directory. This helps both the user and the AI (like Pi.dev) remember useful workflows. Inspired by NVIDIA's Voyager, the system will allow the AI to "save" skills/commands into a library.

## Requirements
1. **Atuin Integration:** Leverage Atuin to store and retrieve shell history, preferring official CLI/API mechanisms over direct database hacking.
2. **AI Command Tagging:** Record AI-generated commands (from Pi.dev's bash execution) into the history, tagged so we know they were generated by the AI.
3. **Success Tracking:** 
   - Not all non-zero exit codes mean a bad command (e.g., a failing test suite is still a correct command).
   - Track genuine bad commands (e.g., bash syntax errors `exit code 2`, command not found `exit code 127`).
   - If the AI immediately corrects a failed command to a successful one, the first was likely bad.
4. **Command Normalization:** Filter or normalize parameters (e.g., `pytest test_foo.py` -> `pytest`) to identify the core commands.
5. **Cheat Sheet UI:** Display a cheat sheet of useful commands. Because a Pi extension only works inside Pi, we should consider a standalone tool/CLI (or Atuin plugin) to display the cheat sheet for the user in any terminal.

## Approach

### 1. MVP: Pi Extension (TypeScript)
- We will start by building a Pi extension (a tool override for `bash` in `.pi/tools/`) written in TypeScript.
- This extension will wrap the normal execution. Before executing a command, it calls `atuin history start "[COMMAND]"`. After execution, it calls `atuin history end <ID> --exit <CODE>`.
- To tag commands as "AI", we can maintain a mapping of Atuin IDs to an "AI-generated" status, or append a special metadata comment to the command before recording it to Atuin.

### 2. The Cheat Sheet & Atuin Integration
- To make the cheat sheet usable outside of Pi, we will create a small CLI tool (written in Go or Rust, or just a shell script/TypeScript binary) that queries Atuin's history for the current working directory.
- This CLI will filter out "bad" commands (syntax errors, typos) and group normalized commands by frequency.

## Files to modify
*MVP - Pi Extension*
- `.pi/tools/bash.ts` - An override for the standard bash tool that integrates `atuin history start/end`.
- `src/analyzer.ts` - Logic to normalize commands and identify true successes vs failures.

*Future - Standalone UI*
- A Go or Rust CLI that queries `atuin search --cwd .` to display the cheat sheet in the user's terminal.

## Questions / Ambiguities
- **Tagging Strategy:** Since Atuin doesn't natively support "tags", should we inject a comment (e.g. `command #ai`) when sending to Atuin, or keep a separate local DB mapping Atuin IDs to the AI tag?
- **Cheat Sheet Tool:** Would you prefer the standalone cheat sheet CLI to be written in Rust or Go? (Both have good SQLite/Atuin integration potential).
